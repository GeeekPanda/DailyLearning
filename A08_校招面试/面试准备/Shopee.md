# 面向对象编程的七大原则
- 开闭原则
    > 软件系统中包含的各种组件，例如模块（Modules）、类（Classes）以及功能（Functions）等等，应该在不修改现有代码的基础上，去扩展新功能。开闭原则中的“开”，是指对于组件功能的扩展是开放的，是允许对其进行功能扩展的；开闭原则中的“闭”，是指对于代码的修改是封闭的，即不应该修改原有的代码。
- 单一职责原则
    > 一个类，只有一个引起它变化的原因。即：应该只有一个职责。
- 里氏替换原则
    > 子类型必须能够替换掉它们的父类型。类B继承类A时，除添加新的方法完成新增功能P2外，尽量不要重写父类A的方法。
- 迪米特法则
    > 一个对象应该对其他对象保持最少的了解。如果两个类不必彼此直接通信，那么这两个类就不应当发生直接的相互作用。
- 依赖倒置原则
    > 高层模块不应该依赖低层模块，二者都应该依赖其抽象；抽象不应该依赖细节；细节应该依赖抽象。中心思想是面向接口编程。
- 接口隔离原则
    > 客户端不应该依赖它不需要的接口；一个类对另一个类的依赖应该建立在最小的接口上。
- 合成/聚成原则
    > 尽量使用合成/聚合，尽量不要使用类继承。换句话说，就是能用合成/聚合的地方，绝不用继承。

# 编译的过程
编译可以分为四个过程：
- 预编译
    源代码文件(.c)和头文件(.h)被预编译器(cpp)预编译成一个.i文件。
    ```shell
    gcc -E hello.c -o hello.i
    cpp hello.c > hello.i
    ```
    预编译做了下列事情：
    - 将所有的#define删除，并展开所有的宏定义；
    - 处理所有的条件预编译指令，比如#if、#ifdef等；
    - 处理#include预编译指令，将被包含的文件插入到该预编译指令的位置。这个过程是递归进行的；
    - 删除所有的注释//和/**/；
    - 添加行号和文件名标识，以便编译时编译器产生调试用的行号信息，以及编译时出现错误或警告可显示行号；
    - 保留所有的#pragma编译器指令，因为编译器需要使用它们。
- 编译
    .i文件被编译器(cc1)编译成汇编代码文件(.s)。
    ```shell
    gcc -S hello.i -o hello.s
    ```
    编译器做了以下事情：
    - 词法分析
        > 将预处理产生的.i文件输入扫描器，扫描器通过算法将源代码的字符序列分割成一系列记号。同时，扫描器还会完成把标识符放到符号表、将数字、、字符串常量放到文字表等工作。
    - 语法分析
        > 语法分析器对由扫描器产生的记号进行语法分析，生成以表达式为节点的语法树。
    - 语义分析
        > 语义分析器完成静态语义分析，通常包括声明和类型的匹配、类型转换等。经过语义分析，语法树的表达式都会被标注上类型，如果有些类型需要做隐式转换，语义分析器会在语法树中插入相应的转换节点。
    - 源代码优化
        > 源代码优化是编译器提供的源代码层级的优化过程，源代码优化器会把整个语法树转换成中间代码，有些值在编译器就可以确定，这类表达式在中间代码就会被优化。
    - 代码生成
        > 将中间代码转换成汇编代码。
    - 目标代码优化
        > 对生成的汇编代码进行优化。
- 汇编
    汇编文件(.s)被汇编器(as)转换成二进制代码可重定位文件(.o)。
    ```shell
    gcc -c hello.s -o hello.o
    as hello.s -o hello.o
    ```
- 链接
    将需要可重定位文件(.o)通过链接器生成最终的可执行文件(a.out)
    ```shell
    ld hello.o xxx.o ...
    ```

# 计算机网络

## TCP与UDP的区别及适用场景
- TCP和UDP区别
    - TCP是面向连接的，即传输数据之前必须先建立连接；UDP是无连接的；
    - 从服务对象看，TCP是点对点的两点间服务，一条TCP连接只能有两个端点；而UDP支持一对一、一对多、多对一、多对多的交互通信。
    - 从可靠性的角度看，TCP是保证可靠性的，即无差错、不丢失、不重复、不乱序；而UDP是尽最大努力交付，不保证可靠性；
    - 从报文长度看，TCP是面向字节流的协议，其报文长度根据接收方的窗口大小和网络拥塞情况决定；而UDP面向报文，不合并不拆分，保留上层协议传下来的报文的边界；
    - 从首部开销看，TCP首部开销大，首部占20个字节；UDP首部开销小，只占8字节。
    - TCP有流量控制和拥塞控制来保证数据传输的安全性；UDP没有拥塞控制，网络拥塞不会影响源主机的发送效率。
- TCP和UDP适用场景
    - TCP是面向连接的可靠的字节流协议，但是其开销大传输速度慢，UDP不可靠无连接但传输速度快，因此选用具体协议时，应根据通信数据的要求而决定。
    - 若对通信数据的完整性要求更高，愿意为了可靠而牺牲一些效率，则应该选用TCP（如FTP、HTTP）；若对通信的实时性要求更高，愿意为效率而牺牲一部分可靠性，那么应该选用UDP（如视频通话、语音通话等）。
    
## TCP的可靠传输机制
可靠性体现在四个方面无差错、不丢失、不重复、不乱序。

通信介质在传输过程中由于电磁干扰或线路问题等原因，会造成比特差错和分组丢失。为了解决比特差错和分组丢失问题，TCP引入了重传机制，要解决比特差错和分组丢失，首先是要检测到这两个问题，检测是否发生比特差错是通过校验和来完成，为检测是否发生分组丢失问题，TCP引入了ACK机制，即接收方收到分组后应给对方发送一个ACK报文段以确认自己收到了分组。如果检测到分组存在比特差错，那么接收方就不给发送方回应ACK，这样过一段时间发送方就会重传这个分组，这样就解决了比特差错和分组丢失问题，但ACK也带来一个问题，即如果ACK丢失的话，发送方重传分组，会导致接收方收到的分组重复，这就引入了重复问题。

为了解决可能发生的分组重复问题，TCP引入序列号机制，每个分组都有一个唯一的序列号，接收方可以通过这个序列号判断他之前是否已经收到了这个分组，如果是，则直接丢弃。从而解决了分组重复问题。

为了提高协议的吞吐量，TCP允许发送方同时发送多个分组，而这就会引入乱序问题，对此，TCP引入了滑动窗口机制
